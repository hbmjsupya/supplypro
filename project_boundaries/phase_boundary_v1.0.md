## 二、开发约束要求

### 2.1 禁止事项
以下行为在未经授权的情况下严格禁止：

1. **核心业务逻辑修改**
   - 禁止修改已稳定模块的核心业务规则
   - 禁止更改状态流转逻辑
   - 禁止修改数据校验规则

2. **数据结构调整**
   - 禁止删除或重命名现有数据库表
   - 禁止删除或重命名现有字段
   - 禁止修改字段的类型和约束
   - 禁止删除或修改现有索引

3. **接口定义变更**
   - 禁止删除现有API接口
   - 禁止修改现有接口的请求参数
   - 禁止修改现有接口的响应结构
   - 禁止更改接口的URL路径

4. **前端交互变更**
   - 禁止修改现有页面的核心交互流程
   - 禁止删除现有功能入口
   - 禁止更改表单的必填项规则

### 2.2 允许事项
以下行为在符合规范的情况下允许：

1. 新增功能模块
2. 新增API接口（不影响现有接口）
3. 新增数据库表和字段
4. 新增前端页面和组件
5. 性能优化（不改变业务逻辑）
6. Bug修复（不改变原有功能行为）

## 三、多问题对话处理规则

### 3.1 多问题判断标准

以下情况视为"包含多个问题"的对话：

1. **显式多问题**
   - 用户消息中包含多个问号（？或?）
   - 用户消息中包含编号列表（1. 2. 3. 或 一、二、三、）
   - 用户消息中包含"同时"、"另外"、"此外"、"还有"等连接词
   - 用户明确要求完成多个独立任务

2. **隐式多问题**
   - 一个功能需求涉及多个模块的修改
   - 一个任务需要前端和后端同时开发
   - 一个需求包含创建、测试、部署等多个阶段
   - 用户请求中包含"全面"、"完整"、"所有"等全量词汇

3. **关联性问题**
   - 主任务完成后需要验证的任务
   - 功能开发后需要生成文档的任务
   - 代码修改后需要运行测试的任务

### 3.2 完成状态验证条件

在提交功能完成确认申请前，必须满足以下所有条件：

1. **问题清单确认**
   - 已识别并记录对话中包含的所有问题
   - 每个问题都有明确的处理状态（待处理/处理中/已完成/已跳过）
   - 跳过的问题必须有用户明确同意的记录

2. **功能完成验证**
   - 所有代码修改已完成并通过编译/构建
   - 所有新增功能已通过基本功能测试
   - 所有修复的Bug已验证不再复现
   - 相关的单元测试/集成测试已通过

3. **文档更新验证**
   - API文档已更新（如有接口变更）
   - 数据库迁移脚本已创建（如有数据结构变更）
   - 用户文档已更新（如有功能变更）

4. **环境一致性验证**
   - 前端服务可正常启动和访问
   - 后端服务可正常启动和响应
   - 数据库连接正常

### 3.3 功能完成确认申请格式

提交功能完成确认申请时，必须使用以下格式：

```markdown
## 功能完成确认申请

### 一、问题清单
| 序号 | 问题描述 | 处理状态 | 完成情况说明 |
|-----|---------|---------|-------------|
| 1 | [问题描述] | 已完成/已跳过 | [具体完成情况] |
| 2 | [问题描述] | 已完成/已跳过 | [具体完成情况] |
| ... | ... | ... | ... |

### 二、完成情况汇总
- **总问题数**: X个
- **已完成数**: X个
- **已跳过数**: X个（需附用户同意记录）
- **待处理数**: 0个（必须为0才能提交）

### 三、关键变更清单
1. **代码变更**
   - [文件路径]: [变更说明]
2. **数据库变更**
   - [迁移脚本]: [变更说明]
3. **配置变更**
   - [配置文件]: [变更说明]

### 四、验证结果
- [ ] 编译/构建通过
- [ ] 功能测试通过
- [ ] 相关测试用例通过
- [ ] 服务可正常启动

### 五、遗留事项
- [如有遗留事项，详细说明原因和处理建议]

### 六、用户确认
请确认以上功能是否满足您的需求。
```

### 3.4 禁止行为

1. **禁止部分完成确认**
   - 不得在存在未处理问题时提交完成确认
   - 不得在用户未明确同意的情况下跳过任何问题
   - 不得将"待处理"状态的问题标记为已完成

2. **禁止模糊确认**
   - 不得使用"基本完成"、"大部分完成"等模糊表述
   - 不得隐瞒或遗漏问题清单中的任何问题
   - 不得在未验证的情况下声称功能已完成

3. **禁止强制确认**
   - 不得在用户提出异议时强制要求确认
   - 不得在测试未通过时要求用户确认
   - 不得在服务异常时要求用户确认

### 3.5 异常处理流程

当遇到以下异常情况时，应按流程处理：

1. **问题无法完成**
   - 详细说明无法完成的原因
   - 提供替代方案或建议
   - 等待用户明确同意后方可跳过

2. **问题需要用户决策**
   - 提供多个可选方案
   - 说明各方案的优缺点
   - 等待用户选择后方可继续

3. **问题之间存在依赖**
   - 明确说明依赖关系
   - 按依赖顺序处理问题
   - 前置问题未完成时不得处理后置问题
