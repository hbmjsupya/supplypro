# SupplyPro 供应链管理系统 - 产品需求文档(PRD)

## 一、产品概述

### 1.1 产品定位
SupplyPro 是面向供应链运营团队的B端管理系统，核心解决商品采购、仓储库存、财务结算三大业务域的数字化管理需求。系统作为供应链中台，对接上游运营平台获取订单数据，对下游供应商和物流商进行业务协同。

### 1.2 目标用户
| 角色 | 职责 |
|------|------|
| 供应链管理员 | 品牌维护、供应商管理、商品池管理 |
| 采购专员 | 采购单创建/发货/收货/入库 |
| 仓库管理员 | 入库确认、出库管理、库存盘点 |
| 财务专员 | 结算单审核/付款、预付款管理、成本票管理 |
| 系统管理员 | 数据维护、系统配置 |

### 1.3 核心价值
- **全流程数字化**: 从采购到结算的完整业务闭环
- **库存精准管理**: 批次化库存、实时结存计算、多维度库存报表
- **财务合规**: 结算审批流、成本票管理、预付款管控
- **物流可视化**: 快递鸟API集成，实时物流轨迹追踪

| 功能 | 优先级 | 描述 |


## 四、业务流转与逻辑说明

### 4.1 采购业务主流程

#### 流转图
```
运营平台推送订单
      ↓
平台待确认订单列表
      ↓ (确认)
生成采购单(待处理)
      ↓ (确认)
采购单(已确认)
      ↓ (发货)
采购单(已发货) → 物流轨迹追踪
      ↓ (收货)
采购单(已收货) → 自动创建入库单
      ↓ (入库确认)
入库单(已入库) → 创建库存批次 + 库存流水
      ↓
采购单出现在待结算列表
      ↓ (发起结算)
创建结算单(待结算)
      ↓ (确认)
结算单(已结算)
      ↓ (付款)
结算单(已付款)
```

#### 关键逻辑说明

**1. 平台订单确认逻辑**
- 平台推送的订单按订单类型(OrderPurchase/Replenishment)生成对应类型的采购单
- 供应商匹配优先级：先按ID精确匹配 → 再按名称模糊匹配 → 最后取第一个可用供应商
- 成本承担方决定初始结算状态：供应商承担→已结算，平台承担→未结算
- 确认时自动创建采购单快照

**2. 采购单发货逻辑**
- 发货方式分为物流配送和自配送两种
- 物流配送：必须选择物流供应商+物流公司+运单号
- 自配送：必须选择物流供应商+配送员+联系电话
- 运单号实时查重，不允许重复
- 物流费用>0时自动创建待结算配送单
- 发货后自动同步物流状态（快递鸟API）
- 乐观锁(@Version)保护并发修改

**3. 采购单收货逻辑**
- 收货后自动创建入库单（关联仓库和采购单）
- 收货后自动创建采购单快照
- 同步更新结算状态
- 收货失败时整个事务回滚

**4. 入库确认逻辑**
- 确认入库时创建库存批次，批次号格式为"日期序号+供应商名"
- 创建入库库存流水(flowType=入库)
- 入库前自动检查是否有已审批的调价单，如有则使用调价后成本
- 更新库存批次的可用数量和总成本

**5. 采购单删除逻辑**
- 只有"待处理"和"已取消"状态的采购单可删除
- 关联入库单状态必须为"待入库"或"已取消"
- 删除时级联删除关联的入库单、库存批次、库存流水

### 4.2 退款业务流程

#### 流转图
```
运营平台推送退款单
      ↓
┌─────────────────────────────────────────┐
│ 仅退款+供应商承担                        │
│   → 自动完成(已完成)                     │
│   → 若采购单待处理则自动取消              │
│   → 出现在待结算列表                     │
├─────────────────────────────────────────┤
│ 仅退款+平台承担                          │
│   → 不推送至本项目(直接拒绝)              │
├─────────────────────────────────────────┤
│ 退货退款                                 │
│   → 有物流单号: 状态直接变为退货中        │
│   → 无物流单号: 状态为待处理              │
│   → 确认收货 → 已完成                    │
│   → 采购单+平台承担 或 出库单:           │
│     创建退货入库单 + 退货入库流水         │
│   → 采购单+供应商承担:                   │
│     不创建入库单                         │
│   → 出现在待结算列表                     │
└─────────────────────────────────────────┘
```

#### 关键逻辑说明

**1. 退款单创建逻辑**
- 仅退款+平台承担：系统直接拒绝创建，抛出异常"仅退款且承担方为平台的退款单不会推送至本项目"
- 仅退款+供应商承担：自动完成，状态直接设为"已完成"；若关联采购单为"待处理"则自动取消采购单
- 退货退款：根据是否有物流单号决定初始状态（有→退货中，无→待处理）

**2. 退款确认收货逻辑**
- 只有"退货中"状态才能确认收货
- 确认收货后的业务分支：
  - 采购单+供应商承担：不创建入库单，退款金额记入待结算列表
  - 采购单+平台承担：创建退货入库单
  - 出库单（不论承担方）：创建退货入库单
- 退货入库单确认后创建退货入库流水(flowType=退货入库)，库存增加

**3. 退款单与结算的关联**
- 退款单完成后出现在待结算列表
- 退款金额为负数，在结算单中冲减采购单金额
- 撤销结算单后，退款单重新回到待结算列表

### 4.3 出库业务流程

#### 流转图
```
创建出库单 → 冻结库存(可用数量减少，冻结数量增加)
      ↓ (发货)
出库单(已发货) → 扣减库存(冻结数量减少，总数量减少) + 创建出库流水
      ↓ (签收)
出库单(已签收)
      ↓
出现在待结算配送列表
      ↓ (发起结算)
创建物流结算单
```

#### 关键逻辑说明

**1. 出库单创建逻辑**
- 选择商品和库存批次，指定出库数量
- 创建时冻结库存：对应批次的可用数量减少，冻结数量增加
- 可发数量 = 可用数量 - 冻结数量，必须≥出库数量
- 出库明细以JSON格式存储在outboundItems字段

**2. 出库单发货逻辑**
- 必须选择物流供应商
- 发货时扣减库存：冻结数量减少，总数量减少
- 创建出库库存流水(flowType=出库)
- 批次可用数量为0时自动标记为"已售罄"

**3. 出库单取消逻辑**
- 只有"待发货"状态可取消
- 取消时释放冻结库存：冻结数量减少，可用数量增加
- 已发货的出库单不可取消，只能通过"重置"操作回退

**4. 出库单来源类型**
- 销售出库(SALES)：关联销售订单，需校验库存充足
- 代发出库(DROPSHIP)：一件代发模式
- 采购出库(PURCHASE)：采购退货出库

### 4.4 结算业务流程

#### 流转图
```
待结算采购单列表 + 待结算配送单列表
      ↓ (勾选+发起结算)
创建结算单(待结算)
      ├── 确认 → 已结算 → 付款 → 已付款
      ├── 撤销 → 已撤回 → 项目回到待结算列表
      └── 驳回 → 已驳回
```

#### 关键逻辑说明

**1. 待结算列表数据来源**
- 已收货且未结算的采购单
- 已完成且未结算的退款单
- 已结算但被撤销的采购单/退款单（重新出现）
- 排除已纳入有效结算单的项目

**2. 结算单创建逻辑**
- 同一供应商的采购单和退款单可合并结算
- 退款金额冲减采购金额，净额为实际应付金额
- 创建结算单后，已勾选项从待结算列表移除
- 结算单号自动生成，每分钟最多999个

**3. 结算单付款逻辑**
- 支持预付款余额抵扣：自动扣减供应商预付款余额
- 预付款余额不足时拒绝付款
- 付款后结算单状态变为"已付款"

**4. 结算单撤销逻辑**
- 撤销后结算单状态变为"已撤回"
- 原结算项（采购单/退款单）重新回到待结算列表
- 采购单和退款单的结算状态重置为"未结算"
- 如使用了预付款余额，撤销后余额不自动退回（需手动处理）

**5. 结算单编辑/删除逻辑**
- 只有"待结算"/"已撤回"/"已驳回"状态可编辑
- "已结算"/"已付款"状态不可删除
- 只有"待结算"状态可删除

### 4.5 预付款业务流程

#### 流转图
```
提交充值申请(待审批) → 审批通过(已审批) → 财务付款(已付款)
                        ↓
                     驳回(已驳回) / 撤回(已撤回)

成本票管理:
上传成本票 → 部分上传/全部上传 → 红冲票(金额≤已上传金额)
```

#### 关键逻辑说明

**1. 预付款充值逻辑**
- 提交充值申请时填写付款方信息和申请金额
- 审批通过后由财务执行付款
- 付款后供应商预付款余额增加，同时创建充值流水(type=充值)
- 充值金额必须为正数

**2. 预付款扣减逻辑**
- 结算单使用预付款余额支付时自动扣减
- 扣减金额必须为正数
- 余额不足时拒绝扣减
- 扣减后创建消费流水(type=消费)

**3. 成本票管理逻辑**
- 成本票状态：未上传 → 部分上传 → 已上传
- 上传金额不能大于未收金额
- 红冲金额不能大于已上传成本票金额
- 已上传金额 ≥ 审批金额时自动更新状态为"已上传"
- 新收到金额 > 0 但 < 成本票金额时更新状态为"部分上传"

**4. 预付款退回逻辑**
- 退回金额必须为正数
- 退回后供应商预付款余额增加
- 创建退回流水(type=退回)

### 4.6 成本调价业务流程

#### 流转图
```
对采购单发起调价 → 调价单(待审批)
      ↓ (审批通过)
更新采购单明细成本 + 更新库存批次成本 + 创建调价流水 + 自动创建结算单
      ↓ (驳回)
调价单(已驳回) → 采购单成本不变
      ↓ (撤销)
调价单(已撤销) → 采购单成本不变
```

#### 关键逻辑说明

**1. 调价单创建逻辑**
- 同一采购单不能存在待审批的调价单（防重复调价）
- 调价后成本不能小于0
- 调价前成本必须与当前采购单明细成本一致

**2. 调价单审批通过逻辑**
- 更新采购单明细的unitCost和totalCost
- 更新采购单的totalAmount
- 入库采购类型：同步更新库存批次的unitCost和totalCost
- 创建调价类型库存流水(flowType=调价)，仅改变成本不改变数量
- 调价差额不为0时自动创建结算单

**3. 调价单驳回/撤销逻辑**
- 只有"待审批"状态可驳回或撤销
- 驳回/撤销后采购单成本不变

### 5.2 消息队列架构（RabbitMQ）

#### 5.2.1 消息拓扑

| 交换机 | 队列 | 路由键 | 死信交换机 | 死信队列 | 说明 |
|--------|------|--------|-----------|---------|------|
| product.exchange | product.es.sync.queue | product.es.sync | product.dlx.exchange | product.dlx.queue | 商品ES同步 |
| purchase.order.snapshot.exchange | purchase.order.snapshot.backfill.queue | purchase.order.snapshot.backfill | purchase.order.snapshot.dlx.exchange | purchase.order.snapshot.dlx.queue | 快照回填 |

**设计要点**：
- 两套独立的死信队列机制，互不干扰
- 所有队列均为 `durable`（持久化），RabbitMQ重启后消息不丢失
- 商品同步队列配置 `x-dead-letter-exchange` 指向专属死信交换机

#### 5.2.2 商品同步流程

```
商品CRUD操作 → ProductSyncProducer发送productId
    → product.es.sync.queue → ProductSyncConsumer消费
    → 查询MySQL商品数据 → 转换为ProductDocument → 写入Elasticsearch
    → 消费失败 → product.dlx.queue → 记录日志/告警
```

**关键设计**：
- `RabbitTemplate` 使用 `@Autowired(required = false)`，开发环境无RabbitMQ时不报错
- 消息体仅为 `Long` 类型的 `productId`，轻量级设计
- `ProductSyncConsumer` 使用 `@Profile("!dev")`，开发环境不激活
- 商品不存在时从ES中删除对应文档

#### 5.2.3 快照回填流程

```
数据修复/历史补录 → SnapshotBackfillProducer发送purchaseOrderId
    → snapshot.backfill.queue → SnapshotBackfillConsumer消费
    → 执行快照回填 → 成功: 手动ACK
    → 失败: 抛出RuntimeException → 触发重试/DLX
    → 最终进入 snapshot.dlx.queue → SnapshotDlxConsumer记录MANUAL INTERVENTION REQUIRED
```

**关键设计**：
- 手动ACK模式（`Channel.basicAck`），确保处理成功后才确认
- 失败后进入独立死信队列，记录需要人工干预的失败任务
- 预留扩展点：可保存到 `failed_jobs` 表或发送告警

- 缺少Supplier或CreatedAt的采购单

- 缺少采购单或仓库关联的入库单

**快照一致性**：
- 实时采购单与最新快照的状态和金额比对
- 创建超过5分钟但仍无快照的采购单
- `snapshot_data`为NULL的无效快照（自动清理）
- 同一采购单多条`is_latest=true`的重复记录

**NULL Product ID检查**（扫描5张表）：
- sales_order_items.product_id
- purchase_order_items.product_id
- inbound_order_items.product_id
- stock_batches.product_id
- stock_flows.product_id

**采购单关联完整性**：
- INBOUND类型且状态为CONFIRMED/SHIPPED但无关联入库单
- 缺少仓库ID的入库单
- 缺少供应商ID的采购单
